Temporal

Temporal dev server ready to include — workflow orchestration with SQLite storage and built-in Web UI, no external deps.

The flox/temporal environment packages Temporal for one-line inclusion in any Flox manifest. Add it via { remote = "flox/temporal" } and the toolchain is pinned, services are wired, and the demo recipe starts cleanly. Published by Flox under MIT license, upstream Temporal, maintained by @rok.

  • Temporal dev server via `temporal-cli` — all-in-one binary
  • gRPC frontend on port 7233, Web UI on port 8233
  • SQLite storage under `$FLOX_ENV_CACHE/temporal`
  • Works with all Temporal SDKs (Python, Go, Java, TypeScript)
  • Boots as a Flox service with `--start-services`
New to Flox? Install it in 30 seconds: curl -L https://downloads.flox.dev/by-env/stable/install.sh | bash — then run the activate command below. Docs →

Recipes

Quick start

temporal-demo
  • Launches Temporal dev server as a Flox service
  • Web UI available at http://127.0.0.1:8233
  • Sample workflow and worker with styled terminal output
composed manifest.toml 
schema-version = "1.10.0"

[include]
environments = [
  { dir = "../temporal" },
  # Brings claude-managed, which auto-discovers the
  # temporal-developer skill installed below from
  # $FLOX_ENV/share/claude-code/skills/temporal-developer/.
  { dir = "../claude" }
]

# Go toolchain for the sample workflow + worker.
# gcc is needed because Go's cgo-enabled stdlib build
# step on first activation calls out to a C compiler.
[install]
go.pkg-path = "go"
go.pkg-group = "go"
gcc.pkg-path = "gcc"
gcc.pkg-group = "go"
# Used by test.sh to parse `temporal task-queue describe`
# JSON output for the worker-readiness check.
jq.pkg-path = "jq"
jq.pkg-group = "demo-tools"
# Demo-only banner UI.
gum.pkg-path = "gum"
gum.pkg-group = "demo-tools"
# Temporal Developer skill, auto-discovered by claude-managed.
skills-temporal-developer.pkg-path = "flox/skills-temporal-developer"
skills-temporal-developer.pkg-group = "skills-temporal-developer"

[hook]
on-activate = '''
# Point Go's caches at a writable location.
# Default GOPATH is $HOME/go, which fails inside the
# CI test namespace where HOME=/var/empty (read-only).
export GOPATH="$FLOX_ENV_CACHE/go"
export GOMODCACHE="$GOPATH/pkg/mod"
export GOCACHE="$FLOX_ENV_CACHE/go-build"
mkdir -p "$GOPATH" "$GOCACHE"

if [[ "${FLOX_ENVS_TESTING:-}" != "1" ]]; then
  gum style --border double --margin '1 2' \
    --padding '1 4' \
    'Start Temporal + worker in the background:' \
    '  flox services start' \
    '  flox activate --start-services' '' \
    'Send a hello workflow:' \
    '  go run ./start YourName' '' \
    'Temporal Web UI:' \
    "  http://$TEMPORAL_HOST:$TEMPORAL_UI_PORT"
fi
'''

[services]
worker.command = '''
  cd "$FLOX_ENV_PROJECT"
  export GOPATH="$FLOX_ENV_CACHE/go"
  export GOMODCACHE="$GOPATH/pkg/mod"
  export GOCACHE="$FLOX_ENV_CACHE/go-build"
  mkdir -p "$GOPATH" "$GOCACHE"
  until temporal workflow list \
      --address "$TEMPORAL_HOST:$TEMPORAL_PORT" \
      >/dev/null 2>&1; do
    sleep 1
  done
  exec go run ./worker
'''

temporal-demo

Full Temporal demo environment with styled terminal output. Includes the Temporal dev server (via the minimal temporal layer) and gum for formatted usage hints on activation.

For a minimal environment to include in your own project, see temporal or use flox/temporal on FloxHub.

Quick start

flox activate -r flox/temporal-demo --start-services
go run ./start World

You should see Workflow result: Hello World printed by the starter. The Temporal Web UI is at http://127.0.0.1:8233.

What this demo includes

  • Temporal dev server with built-in Web UI (inherited from the temporal minimal env)
  • A Go worker (worker/main.go) running as a flox service on my-task-queue
  • A starter binary (start/main.go) that triggers a SayHelloWorkflow
  • A trivial greeting Go package showing the workflow/activity split
  • gum for the activation banner
  • jq for the test harness’s worker-readiness probe

Web UI

After starting services, open the Temporal Web UI:

http://localhost:8233

The Web UI lets you browse workflows, view execution history, and inspect namespace configuration.

CLI examples

List registered namespaces:

temporal operator namespace list

List workflows in the default namespace:

temporal workflow list

Describe a specific workflow execution:

temporal workflow describe --workflow-id <id>

SettingValue
Host127.0.0.1

Customizing connection settings

Override vars in your own manifest:

[vars]
TEMPORAL_PORT = "17233"
TEMPORAL_UI_PORT = "18233"
TEMPORAL_HOST = "0.0.0.0"

Data directory

Temporal data is stored in $FLOX_ENV_CACHE/temporal. This persists across activations. To start fresh:

rm -rf "$FLOX_ENV_CACHE/temporal"

Using the minimal version

If you do not need the demo experience, include the minimal environment in your own manifest:

[include]
environments = [{ remote = "flox/temporal" }]

This gives you the Temporal dev server with sane defaults. Override vars and add packages in your own manifest as needed.

Service management

flox services start            # start temporal
flox services stop             # stop temporal
flox services status           # check status
flox services logs temporal    # view logs

Documentation

temporal

Temporal dev server with SQLite storage and built-in Web UI. Bundles two Flox-packaged binaries:

  • temporal-cli (cmd temporal) — workflow/namespace management plus an embedded dev server with Web UI. Drives the [services] block below.
  • temporal server (cmd temporal-server, plus temporal-sql-tool, temporal-cassandra-tool, tdbg) — production binaries for running against PostgreSQL/MySQL/Cassandra clusters.

No external dependencies required for dev use.

Usage

Include this environment in your own manifest:

[include]
environments = [{ remote = "flox/temporal" }]

Then start the service:

flox activate
flox services start

Configuration

Override these variables in your manifest as needed:

VariableDefaultDescription
TEMPORAL_HOST127.0.0.1Bind address
TEMPORAL_PORT7233gRPC frontend port
TEMPORAL_UI_PORT8233Web UI port

Endpoints

  • Web UI: http://127.0.0.1:8233
  • gRPC (for SDKs): 127.0.0.1:7233

Connect your Temporal SDK client to the gRPC endpoint:

client = await Client.connect("127.0.0.1:7233")
c, err := client.Dial(client.Options{
    HostPort: "127.0.0.1:7233",
})

Production server

For non-dev use, temporal-server is available as well. It speaks the same protocol but needs an external database. Point it at a config file:

temporal-server start \
  --config /path/to/config \
  --service frontend,history,matching,worker

temporal-sql-tool and temporal-cassandra-tool handle schema setup; tdbg is the debugger.
