Honcho

Self-hostable memory infrastructure for stateful agents — FastAPI server, background deriver, postgres+pgvector, redis, and the Python SDK in one flox env.

flox/honcho packages the open-source Honcho server (the Plastic Labs memory layer for stateful agents) with all of its runtime dependencies sealed into one Nix closure. The env ships the FastAPI app, the background `deriver` worker that builds peer representations, and the `honcho-ai` Python SDK — plus a local PostgreSQL (with pgvector) and Redis wired up as flox services so a single `flox activate --start-services` gets you a running stack. Set an LLM provider key (OpenAI / Anthropic / Gemini) and the reasoning endpoints come alive too.

  • Run Honcho locally — FastAPI server + deriver worker
  • Bundled honcho-ai SDK, no `pip install` needed
  • PostgreSQL + pgvector + Redis as flox services
  • Alembic migrations applied on first activate

Honcho Demo

Interactive demo for the self-hosted Honcho memory server. Composes the honcho minimal env (server + deriver

  • postgres + redis + SDK) and ships a quickstart.py that exercises the full Store → Reason → Query → Inject loop.

Try it

cd honcho-demo
flox activate --start-services
# Optional: enables the reasoning path
export LLM_OPENAI_API_KEY=sk-...
python quickstart.py

You should see four messages stored on demo-session, then either a peer chat answer (if a key was set) or a clean skip.

What you get

Source envWhat it adds
../honchoserver, deriver, SDK, postgres+pgvector, redis
demo-toolsgum for the banner
(this env)quickstart.py sample script

What quickstart.py does

1. Connects to http://127.0.0.1:18000 (override with $HONCHO_URL)
2. Creates a workspace, two peers (alice + tutor), a session
3. Adds four messages (storage path, no LLM key needed)
4. Calls alice.representation() (storage path, no LLM key)
5. If an LLM key is set:
   - alice.chat("What is the user struggling with?")
   - session.context(summary=True, tokens=2000).to_openai(...)

The script’s loop mirrors the upstream README quickstart exactly, just pointed at the local server instead of api.honcho.dev.

Wiring it into your own code

The SDK and CLI are on PATH already:

from honcho import Honcho

h = Honcho(
    workspace_id="my-app",
    base_url="http://127.0.0.1:18000",
    api_key="local-dev",   # AUTH_USE_AUTH=false in the minimal env
)
peer = h.peer("user-42")
session = h.session("convo-1")
session.add_messages([peer.message("hello")])

Documentation

Honcho

Self-hostable Honcho environment — memory infrastructure for stateful agents. Ships the FastAPI server, the background deriver worker, the honcho-ai Python SDK, and local PostgreSQL (with pgvector) + Redis wired up as flox services.

Quick start

Include in your manifest:

[include]
environments = [{ remote = "flox/honcho" }]

Then activate and start the services:

flox activate --start-services

The Honcho API listens on http://127.0.0.1:18000 and /health returns 200 once it’s ready. The Python SDK is importable straight away:

from honcho import Honcho
honcho = Honcho(
    workspace_id="my-app",
    base_url="http://127.0.0.1:18000",
    api_key="local-dev",
)

What it provides

honcho-serverRuns the FastAPI app via uvicorn
honcho-deriverRuns the background reasoning worker

Plus the honcho-ai Python SDK in the bundled venv — from honcho import Honcho works without pip install.

Honcho’s reasoning endpoints (chat, representations) need at least one LLM provider key. Set whichever you have:

export LLM_OPENAI_API_KEY=sk-...
# or LLM_ANTHROPIC_API_KEY, LLM_GEMINI_API_KEY

Storage endpoints (workspaces, peers, sessions, messages, /health) work without an LLM key.

See Honcho’s configuration reference for the full set of env vars and TOML sections.

How data is stored

All state lives under $FLOX_ENV_CACHE/honcho/:

  • pgdata/ — PostgreSQL cluster (initialized + migrated on first activate)
  • redis/ — Redis dump
  • logs/init.log — only present if first-time init failed

flox delete wipes the cache.
